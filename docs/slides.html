从 Spring MVC 到 Spring Boot

演进之路 + demo1 · ~35min

1.1 快速回顾（上节课）

  • Spring = IoC 容器，管理对象的创建和注入

1.2 先看结果——demo1-initializer

四个文件，零行 XML，0.4 秒启动，IoC 和 AOP 全在工作。

// Demo1Application.java —— 构造器注入 + 启动后自动执行
@SpringBootApplication
public class Demo1Application implements CommandLineRunner {
    private final GreetingService greetingService;
    public Demo1Application(GreetingService greetingService) {
        this.greetingService = greetingService;
    }
    public static void main(String[] args) { SpringApplication.run(Demo1Application.class, args); }
    public void run(String... args) { System.out.println(greetingService.greet("产品团队")); }
}

// ChineseGreetingService.java —— @Service 标记，Spring 自动发现
@Service
public class ChineseGreetingService implements GreetingService {
    public String greet(String name) { return "你好，" + name + "！"; }
}

// LoggingAspect.java —— @Aspect，所有 service 方法自动加计时日志
@Aspect @Component
public class LoggingAspect {
    @Around("execution(* com.example.demo1.service..*.*(..))")
    public Object log(ProceedingJoinPoint jp) throws Throwable {
        long s = System.currentTimeMillis(); Object r = jp.proceed();
        System.out.println("[AOP] " + jp.getSignature().toShortString() + " 耗时:" + (System.currentTimeMillis()-s) + "ms");
        return r;
    }
}
mvn spring-boot:run
你好，产品团队！欢迎来到 Spring Boot 的世界。
[AOP] greet 执行耗时: 2ms

1.3 对比——传统方式需要写什么

传统 Spring（需要 XML）

<context:component-scan base-package="com.example.service"/>
<context:annotation-config/>

每个项目都要写，包名变了得跟着改

Spring Boot（demo1）

0 行 XML。@SpringBootApplication 默认从启动类所在包开始扫描，@Autowired 自动生效。

web.xml、spring-mvc.xml 那 80+ 行是 Web 层配置，demo1 没有 Web 模块。第三章加了 Web Starter 之后再对比。

1.4 "约定大于配置"

配置 = 你告诉框架每一个细节。数据源类名、端口号、扫描路径——全部自己说明。

约定 = 框架预设了默认行为。你按规矩来，细节不用说明。不搞特殊就自动生效。

引入 Spring Boot，不写端口配置。启动后 8080 已在监听。想改？server.port=9090 一行。没写？默认 8080。
把每个人都要写一遍的东西变成默认就有的。想改可以改，不改就能用。

1.5 Spring Boot 是什么

痛点解法
手写 XML 配置自动配置 — 约定好了，不用写
6~8 个依赖版本对齐Starter 套餐 — 一个依赖一组功能
装 Tomcat、打 WAR、部署内嵌 Tomcat — java -jar 直接跑
搭环境半天Spring Initializr — 30 秒生成项目
Spring Boot = Spring + Spring MVC + 自动配置 + 内嵌服务器。它是 Spring 的"自动挡"——手动挡技能全有效，只是不用踩离合。

2.1 @SpringBootApplication 三合一

@SpringBootApplication
public class Demo1Application {
    public static void main(String[] args) { SpringApplication.run(Demo1Application.class, args); }
}
注解作用上节课
@Configuration标记配置类，可定义 Bean涉及
@ComponentScan扫描当前包及子包下组件涉及
@EnableAutoConfiguration自动配置——Spring Boot 的灵魂本节课新内容

2.2 自动配置——两条约定

约定一：默认扫描位置。@ComponentScan 从启动类所在包开始，向下扫描所有子包。Demo1Application 在 com.example.demo1，service/、aspect/ 下的 @Service、@Aspect 自动被发现。类放到包外则扫不到，需显式配置。

约定二：注解自动生效。传统 Spring 需要 <context:annotation-config/> 开启注解处理。Spring Boot 自动开启——写 @Autowired 即生效，无需 XML 开关。

两条约定合在一起：Spring Boot 知道去哪里扫描、扫到之后怎么处理。开发者只需写注解。

2.3 精装修公寓

传统 Spring = 毛坯房

墙自己刷、电线自己铺、地板自己装。好处是想怎么装怎么装，坏处是每个项目都要从头装一遍。

Spring Boot = 精装修公寓

拎包入住。不满意的地方可以改——你的配置优先级高于默认配置。条件装配规则：你需要+你没配→我来；你不需要/你配了→我靠边。

3.2 传统 Spring MVC 需要写什么

同样的 Web 能力，传统方式需要 3 个 XML 文件，80+ 行配置，再加安装 Tomcat、打 WAR 包、部署到 webapps、重启。

// ① web.xml（~30行）：配置 DispatcherServlet
<web-app><servlet><servlet-name>dispatcher</servlet-name>
<servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class>
<init-param><param-name>contextConfigLocation</param-name>
<param-value>/WEB-INF/spring-mvc.xml</param-value></init-param>
</servlet><servlet-mapping><servlet-name>dispatcher</servlet-name>
<url-pattern>/</url-pattern></servlet-mapping></web-app>

// ② applicationContext.xml（~20行）：数据源、事务、扫描
<beans><context:component-scan base-package="com.example"/>
<bean id="dataSource" class="...BasicDataSource">
<property name="driverClassName" value="com.mysql.cj.jdbc.Driver"/>
<property name="url" value="jdbc:mysql://localhost:3306/demo"/></bean>
<bean id="txManager" class="...DataSourceTransactionManager">
<property name="dataSource" ref="dataSource"/></bean></beans>

// ③ spring-mvc.xml（~30行）：视图解析器、注解驱动
<beans><mvc:annotation-driven/>
<context:component-scan base-package="com.example.controller"/>
<bean class="...InternalResourceViewResolver">
<property name="prefix" value="/WEB-INF/views/"/>
<property name="suffix" value=".jsp"/></bean></beans>

而 demo2：一个 starter-web 依赖 + 一个 @RestController。零行 XML。

3.3 承上启下对照表

上节课（传统 Spring MVC）Spring Boot还需要吗
web.xml 配置 DispatcherServletstarter-web 自动配置不需要
spring-mvc.xml 配置视图解析器、注解驱动WebMvcAutoConfiguration 自动配置不需要
applicationContext.xml 配置组件扫描@SpringBootApplication 自带扫描不需要
写 @Controller + @RequestMapping写 @RestController + @GetMapping写法一样
装 Tomcat → 打 WAR → 部署内嵌 Tomcat，java -jar 直接跑不需要
上节课教的 @Controller 怎么用——照用。web.xml 怎么配——不用了。Starter 已经替代了那三个 XML 文件。

3.4 第一个 REST API

@RestController  // = @Controller + @ResponseBody，返回值自动转 JSON
public class GreetingController {
    private final GreetingService greetingService;  // 构造器注入，复用 demo1 的 Service

    @GetMapping("/api/greeting")
    public Result<String> greet(@RequestParam(defaultValue = "世界") String name) {
        return Result.success(greetingService.greet(name));
    }
}

@RestController = 上节课的 @Controller + 自动 JSON 序列化。@GetMapping = @RequestMapping 的 GET 限定版。

3.5 统一响应 Result<T>

问题：每个接口返回格式不一致——有的返回对象，有的返回字符串，有的返回空 Body。前端无法统一处理。

{"code": 200, "message": "操作成功", "data": {...}}

三个字段：code（状态码，200=成功）、message（提示信息）、data（实际数据，泛型）。前端只需判断 code === 200。

3.6 统一异常处理

问题：用户请求不存在的资源时如何处理？Controller 里写满 try-catch？

@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BusinessException.class)
    public Result<?> handle(BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    }
}

@RestControllerAdvice 一个注解，全局处理所有异常。Controller 里不需要任何 try-catch。上节课：每个方法写 try-catch 或在 web.xml 配 error-page。现在：一个类全局完成。

3.7 运行验证

cd demo2-web && mvn spring-boot:run
# 日志关键行：Tomcat started on port 8080

curl 'http://localhost:8080/api/greeting?name=产品团队'
# → {"code":200,"message":"操作成功","data":"你好，产品团队！"}

Tomcat 自动启动，不需要安装和配置。一个依赖加一个 Controller，控制台程序变为 Web 服务。

数据层实战 + demo3-data

~30min

4.1 demo3 比 demo2 多了两个依赖

pom.xml 多了 mybatis-plus-spring-boot3-starter 和 h2。H2 是内存数据库——无需安装 MySQL、无需建表，应用停止数据自动清空。生产环境换 MySQL 只需修改 application.yml 中几行配置，代码不变。

4.2 MyBatis Plus —— 数据层的约定大于配置

// Entity —— 一个类对应一张表
@Data @TableName("employee")
public class Employee {
    @TableId(type = IdType.AUTO) private Long id;
    private String name;
    private BigDecimal salary;         // 金额用 BigDecimal，不用 float
    private LocalDateTime createTime;  // 日期用 LocalDateTime，不用 Date
}

// Mapper —— 空接口 = 全套 CRUD
@Mapper
public interface EmployeeMapper extends BaseMapper<Employee> {
    // 空接口！继承 BaseMapper 自动拥有：insert、deleteById、
    // updateById、selectById、selectList、selectPage、selectCount...
}
传统 MyBatis：每张表约 10 条 SQL，十张表 100 条。MyBatis Plus：继承一个接口，完事。

4.3 条件查询与分页

LambdaQueryWrapper<Employee> wrapper = new LambdaQueryWrapper<>();
wrapper.like(name != null, Employee::getName, name)
       .eq(departmentId != null, Employee::getDepartmentId, departmentId)
       .orderByDesc(Employee::getId);

Page<Employee> page = new Page<>(1, 10);
employeeMapper.selectPage(page, wrapper);
// 返回对象包含：records（数据列表）、total（总数）、current（当前页）、pages（总页数）

LambdaQueryWrapper 链式调用，字段名以方法引用书写——写错时编译器直接报错。分页的 count 查询和 limit 拼接全部自动完成。

4.4 配置管理——XML 与 YAML 对比

spring:
  datasource:
    url: jdbc:h2:mem:demo3;MODE=MYSQL
    username: sa
  sql.init.mode: always

5 行 YAML。传统 Spring 同一配置需约 21 行 XML（bean 定义 + property 标签）。换 MySQL 改 3 行，代码不动。多环境：application-dev.yml / application-prod.yml，一行切换。

04

从单体到微服务

为什么需要 Spring Cloud · ~12min

5.1 单体架构的困境

创业初期：一个 shop.jar 包含订单、用户、商品、库存。java -jar 即上线。

团队 200 人后：

  • 200 人同时改一个工程，合并冲突频繁
  • 订单扩容→整个系统部署多份，不需要扩容的模块也重复部署
  • 改一行代码→整个系统重新部署
  • 一个模块内存泄漏→全站不可用

5.2 微服务的思路——拆分

将一个超级工程拆分为几十个小工程。每个独立开发、独立部署、独立扩容。

类比：单体像所有人一个大开间，一人感冒全传染。微服务像按部门分楼层——技术部三楼、产品部五楼、市场部八楼，各管各的。

5.3 拆分后的新问题

  • 订单服务要调用户服务——怎么知道它在哪台机器上？
  • 前端要调多个服务——难道记住所有地址？
  • 用户服务宕机，订单不停重试→级联故障，全站崩溃？
  • 一个请求跨多个服务——哪一步慢了？
Spring Boot 让一个服务跑起来。Spring Cloud 让一组服务能协作。
05

Spring Cloud 六大组件

~28min · 贯穿比喻：外卖平台

6.1 Nacos 注册中心——配送调度中心

场景：用户下单后，订单服务要通知配送服务找骑手。配送服务跑在 3 台机器上，订单服务怎么知道该找哪台？写死 IP 则扩容时地址变化需要改配置重启。

解法：配送服务启动时向 Nacos 登记地址，订单服务去 Nacos 查询可用实例。机器宕机自动剔除，新增自动加入。调用方不感知变化。

骑手上线打卡→平台知道谁在线。下单→平台自动分配空闲骑手。Nacos = 配送调度中心。

6.2 Gateway 网关——外卖 App 统一后台

场景：App 首页调商家服务、购物车调订单服务、结算调支付服务。让 App 记多个地址？每个服务各自做登录验证——重复且不一致。

解法：App 只与 Gateway 通信。Gateway 按路径转发到对应服务。登录验证统一在 Gateway 完成——没 token 直接拦截。

你不会直接连商家数据库和支付系统——你只跟 App 后台交互。Gateway = 这个统一后台。

6.3 Nacos Config 配置中心——商家后台改价

场景：配送费规则调整，订单服务和配送服务都在用这个规则。需要改两个配置、重启两个服务。漏改一个即线上不一致。

解法：规则统一放配置中心。运营改一次，所有服务自动推送，不重启生效。

商家后台改菜价→App 立刻显示新价格。配置中心 = 这个后台管理系统。

6.4 Feign 服务调用——一键支付

场景：提交订单后要依次调用户服务（验证地址）、商家服务（确认库存）、支付服务（扣款）。每次手写 HTTP 调用十几行，重复数十遍。

解法：声明接口 + @FeignClient 注解 = 自动发 HTTP。像调本地方法一样调远程服务。URL、序列化、超时全部自动处理。

App 点"微信支付"→自动调起微信，不用自己打开微信输金额。Feign = 一键调用。

6.5 Sentinel 熔断降级——支付宕机兜底

场景：支付服务宕机。订单服务不停重试，1000 个请求阻塞等待。订单服务被拖垮→商家服务跟着宕机→级联故障，全站崩溃。商家正常营业、骑手在线，但 App 白屏。

解法：监控调用成功率。支付连续失败→熔断（切断调用）+降级（切换货到付款）。支付恢复→自动切回在线支付。

一家店停电关门→平台暂时下架。Sentinel = 自动下架并推荐替代方案的机制。

6.6 Sleuth/Zipkin 链路追踪——外卖物流追踪

场景：用户投诉"下单等了 10 秒"。请求跨 5 个服务——哪个环节耗时过长？每个服务有自己的日志，但没有关联，拼不出完整链路。

解法：每个请求分配全局唯一 TraceID（快递单号），在服务间传递。Zipkin 界面拼接完整调用树，每步耗时一目了然。

TraceID: order-001
  Gateway      →   5ms ✓
  订单服务      →  20ms ✓
  用户服务      →  30ms ✓
  商家服务      → 8000ms 🚨  查库存 SQL 缺索引
  支付服务      →  15ms ✓
快递物流："到达分拨中心 10:00，离开 18:00"——卡了 8 小时。Zipkin = 每个请求的快递单号。

6.7 六大组件速查——回到外卖平台

外卖场景组件作用
骑手上线打卡，平台自动分配Nacos配送调度中心
App 统一后台，不管后面几个服务Gateway外卖 App 后台
商家改配送费，App 立刻生效Nacos Config商家后台改价
一键支付，不用自己打开微信Feign一键支付
支付宕机，自动切换货到付款Sentinel故障自动兜底

06

产品视角

微服务不是免费的午餐 · ~8min

7.1 拆分的代价

收益成本
独立部署运维复杂度 × N。日志、监控、部署流水线从 1 变 20
独立扩容分布式事务复杂度增加。原来一个数据库事务→现在需协调多个服务
团队自治网络调用引入延迟。本地调用 1ms→HTTP 调用 10ms 起步

微服务不是银弹——它解决了一些问题，同时制造了新的问题。

7.2 拆分前的三个问题

  1. 有独立的业务边界吗？ 不是功能多就该拆。订单和订单详情属同一边界，拆分增加耦合。
  2. 需要独立扩容吗？ 双十一仅订单要加机器——才值得拆。没有独立扩容需求的模块，拆分没有核心收益。
  3. 有独立团队维护吗？ 一个团队管 3 个微服务 = 分布式单体，比单体更复杂却没得到好处。
每拆分一个服务，运维成本就增加一份。拆分之前，先回答这三个问题。
