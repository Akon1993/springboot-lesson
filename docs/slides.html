01

从 Spring MVC 到 Spring Boot

演进之路 · ~15min

📦 传统 Spring 开发有多痛？

😫 四大痛点

配置地狱：web.xml + applicationContext.xml + spring-mvc.xml

依赖管理：6~8 个依赖，版本自己对齐

部署麻烦：装 Tomcat → 打 WAR → 部署

启动慢：搭环境半天起步

还没写代码，100+ 行 XML 先写出来

1. web.xml (30行) → 2. applicationContext.xml (20行) → 3. spring-mvc.xml (30行) → 4. 数据源 → 5. 事务 → 6. 装Tomcat → 7. 打WAR → 8. 部署

💡 Spring Boot 的答案

约定大于配置 — Convention over Configuration

配置 =

你告诉框架每一个细节怎么做

约定 =

你按规矩来，剩下的我自动帮你做

📐 "约定大于配置"三个例子

你的动作Spring Boot 的约定省掉了什么
引入 spring-webmvc"你要用 Spring MVC"→ 自动配 DispatcherServletweb.xml + spring-mvc.xml
继承 BaseMapper<Employee>"你要 CRUD 这个表"→ 自动生成 SQL百行 XML 映射文件
引入 starter-web，不配端口"默认 8080，内嵌 Tomcat"装 Tomcat、配端口、打 WAR
把每次都要写的东西变成默认就有的东西。想改可以改，不改就能用。

Spring Boot =

Spring + Spring MVC + 自动配置 + 内嵌服务器

它不是替代 Spring，它是 Spring 的 "自动挡"

04

MyBatis Plus

数据访问层的"自动挡" · ~15min

🎁 继承即拥有

// Entity —— 一个类对应一张表
@Data
@TableName("employee")
public class Employee {
    @TableId(type = IdType.AUTO)
    private Long id;
    private String name;
    private Integer age;
}

// Mapper —— 继承 BaseMapper，自动获得 CRUD
@Mapper
public interface EmployeeMapper extends BaseMapper<Employee> {
    // 空接口！但拥有 insert/delete/update/select...
}
一张新表 = 一个空接口。完事。

🔗 LambdaQueryWrapper —— 不写 SQL

LambdaQueryWrapper<Employee> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Employee::getName, "张三")     // WHERE name = '张三'
       .gt(Employee::getAge, 25)          // AND age > 25
       .orderByDesc(Employee::getId);     // ORDER BY id DESC

employeeMapper.selectList(wrapper);

字段写错了？编译器直接报错——不是运行时炸

📄 分页——一行搞定

Page<Employee> page = new Page<>(1, 10);
employeeMapper.selectPage(page, wrapper);
// 自动返回：records、total、current、pages

演进路径：JDBC 手写 → MyBatis 半自动 → MyBatis Plus 更自动

06

统一响应与异常处理

前端同事的救命稻草 · ~10min

😱 AI 生成的各种返回

return employee;             // {"id":1,"name":"张三"}
return "success";            // "success"
return ResponseEntity.ok();  // 200 OK 空 Body

四种接口，四种格式。前端疯了。

✅ 统一用 Result<T> 包装

{"code": 200, "message": "操作成功", "data": {...}}
场景返回
成功Result.success(data)
失败Result.error("员工不存在")

前端只要判断 code === 200

🛡️ 全局异常处理

@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BusinessException.class)
    public Result<?> handle(BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    }
}
// Controller 里不需要写任何 try-catch

审 AI 代码三件事：Result 包装？抛 BusinessException？全局 ExceptionHandler？

📝 Prompt 模板

使用 Spring Boot 3.x + MyBatis Plus 开发【功能描述】。

要求：
- 分层架构：Controller / Service / ServiceImpl / Mapper / Entity
- 统一返回 Result<T>
- 增删改加 @Transactional
- 用 LambdaQueryWrapper，不要字符串拼 SQL
- 日期用 LocalDateTime，金额用 BigDecimal
- 异常用 BusinessException，不要空 catch
- 包名用 jakarta，不要 javax
- 不要生成 XML 配置，全用注解和 YAML

每一个要求前面几章都讲过了。你每多说一句，就少踩一个坑。

🔍 15 条审查清单（1/3）

① 文件在正确包下？
② Controller 有 @RestController？
③ Service 接口+实现分离？
④ Mapper 继承 BaseMapper？
⑤ @TableName 对应表名？

你不会写 Java，但你会看有没有这些注解。

🔍 15 条审查清单（2/3）

⑥ 返回值是 Result<T>？
⑦ 增删改有 @Transactional？
⑧ 密码硬编码了？
⑨ 用了 LambdaQueryWrapper？
⑩ catch 块是空的？

🔍 15 条审查清单（3/3）

⑪ 分页用了 Page<T>？
⑫ 日期用 LocalDateTime？
⑬ 参数有 @Valid？
⑭ 日志用 @Slf4j？
⑮ import 有 javax？(应为 jakarta)
这 15 条，不需要会写代码就能查

🐛 AI 最常犯的 3 个错误

🔄

循环里查数据库

for 循环里 selectById → 应该 selectBatchIds

💥

忘了 @Transactional

转账扣钱成功加钱失败 → 钱消失了

🕳️

空 catch

catch(Exception e){} → 定时炸弹

10

Spring Cloud 六大组件

拆完之后的六道保险 · ~28min

📒 ① 注册中心 Nacos —「公司通讯录」

问题

订单服务怎么知道用户服务在哪台机器？写死 IP？扩容就炸。

解法

每个服务启动去 Nacos 登记，调用方去 Nacos 查询

你换了手机号，更新公司通讯录就行。不用群发通知 500 人。

🏢 ② API 网关 Gateway —「写字楼前台」

问题

前端记 30 个地址？每个服务都要做鉴权？

解法

前端只调 Gateway，统一入口 + 统一鉴权。没 token 的直接拦门口。

你去写字楼，前台帮你刷卡、告诉你几楼。你不用自己找电梯。

☁️ ③ 配置中心 Nacos Config —「iCloud 同步」

问题

数据库地址变了 → 改 10 份配置、重启 10 个服务

解法

一处修改，全部自动推送，不重启生效

换了新手机，登录 iCloud——通讯录、壁纸、WiFi 全部自动同步。

📞 ④ 服务调用 Feign —「快捷拨号」

问题

手写 HTTP：拼 URL、发请求、解析响应 — 每个调用十几行

解法

写接口 + 注解 = 自动发 HTTP。像调本地方法一样

设一个"老婆"快捷拨号。说名字就拨号，不用管号码怎么拨的。

⚡ ⑤ 熔断降级 Sentinel —「保险丝」

问题

用户服务挂了 → 订单不停重试 → 订单也挂了 → 级联故障，全站崩溃

解法

发现下游挂了 → 熔断（切断调用）+ 降级（返回兜底结果）

电饭煲短路 → 保险丝熔断。只断厨房一路电，客厅灯还亮着。

📦 ⑥ 链路追踪 Sleuth/Zipkin —「快递追踪」

问题

请求跨 5 个服务，慢了——哪一步的问题？日志散落各处，拼不起来。

解法

每个请求一个 TraceID（快递单号），拼接完整调用树，每步耗时一目了然

快递慢了？一看物流轨迹——在分拨中心卡了 5 小时。破案。

产品视角

微服务不是免费的午餐 · ~8min

⚖️ 拆服务的代价

好处代价
独立部署运维复杂度 × N
独立扩容分布式事务，复杂 10 倍
团队自治HTTP 调用有延迟+抖动

❓ 拆之前三问

① 有独立的业务边界吗？
不是"功能多"就该拆。订单和订单详情是同一个边界。

② 需要独立扩容吗？
双十一只有订单要加机器，用户不用——才值得拆。

③ 有独立团队维护吗？
一个团队维护 3 个微服务 = 分布式单体，更复杂。

每拆一个服务，运维成本就多一份。拆之前，先问这三个问题。
